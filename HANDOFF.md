## Хто клієнт і навіщо це

Власниця сайту (продає запчастини для машинок для стрижки Wahl/Moser/
Andis) хоче, щоб Google міг індексувати кожен товар/категорію/бренд
окремо. Зараз сайт — SPA (один HTML, все генерується JS), Google без
виконання JS бачить порожню сторінку, і навіть з виконанням JS немає
окремих URL, sitemap, унікальних Title/Description.

## РЕАЛЬНА СТРУКТУРА ТАБЛИЦІ (підтверджено скріншотами клієнтки 22.07, вже вбудовано в код)

Таблиця "Liga kra..." має 4 вкладки:

- **Products**: id, name, sku, brand, price, cat, model, inStock, emoji,
  img, desc, oldPrice, features, top, createdAt
  - `cat` — це КОД категорії (напр. "blades"), не текст
  - `model` — це КОД моделі (напр. "m17"), не текст
- **Categories**: id, label, icon (напр. id="blades", label="ножі та ножові блоки")
- **Models**: id, name, brand, emoji, img (напр. id="m17", name="T-Cut", brand="Moser")
- **Orders**: замовлення (не чіпали, свій API `/api/order` з Етапу 3 не залежить від цієї вкладки)

`lib/sheets.js` УЖЕ переписаний під цю структуру (з попереднього "гнучкого
вгадування" полів на конкретну схему): об'єднує Products з Categories і
Models (join за id), показує людські назви на сайті, зберігає коди для
запису назад. Slug сторінки товару генерується з бренду й артикула (sku),
а не з назви — стабільніше, бо назви товарів можуть повторюватись.
Форма адмінки (`ProductForm`) тепер має dropdown для категорії й моделі
замість вільного тексту.

### ДВІ РЕЧІ, ЯКІ Я ЩЕ НЕ МІГ ПЕРЕВІРИТИ (клієнтка не давала код Apps Script)

1. **Формат відповіді SHEET_API_URL** — я ПРИПУСКАЮ, що один запит
   повертає `{ products: [...], categories: [...], models: [...] }`
   одним JSON (функція `fetchAllSheets()` у `lib/sheets.js`). Якщо
   насправді там окремі endpoint'и чи параметр `?sheet=Products` —
   треба підправити ЛИШЕ цю функцію, решта коду не залежить від деталей.
2. **Формат запису (POST) для адмінки** — так само припущення
   (`{ action: 'save', product: {...} }`). Треба звірити з реальним
   кодом Apps Script клієнтки (Code.gs).

Якщо продовжуєш цю розмову і клієнтка ще не скинула код Apps Script —
попроси його (Code.gs або хоча б відповідь SHEET_API_URL як текст/JSON),
щоб зняти обидва припущення, перш ніж запускати сайт наживо.

  зовнішнього деплою, тому технічно можна, але для консистентності й
  можливості показувати компоненти як прев'ю — уникати, використовувати
  React-стан або cookies/Server Actions)
- Усі назви полів з Google Sheets — гнучкі (див. normalizeProduct у
  lib/sheets.js), бо реальна структура таблиці клієнта ще не звірена.
  Якщо додаєш нові поля (наприклад, для замовлень) — одразу лишай
  коментар у стилі "підлаштуйте назву колонки під вашу таблицю"
- Клієнт — не технічна людина. Пояснення в чаті — простою мовою,
  без надлишкового жаргону, з конкретними кроками "що натиснути"
- Якщо для наступного кроку критично не вистачає одного факту (напр.
  ключ API, назва таблиці замовлень) — постав ОДНЕ конкретне питання,
  не блокуй решту роботи, продовжуй усе, що можна зробити без відповіді

файл і почне працювати без додаткових уточнень, крім, можливо, одного
питання про Telegram/збереження замовлень, якщо це не буде очевидно.
